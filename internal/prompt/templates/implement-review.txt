## Review Instructions

First, examine the current state of the codebase. Prefer `git diff` to see what has changed.
If git is unavailable or this is not a git repository, do **not** skip review. Continue by inspecting available files/artifacts directly and still produce a full structured review.

Then evaluate the implementation against these criteria:

1. **Plan Adherence**: Does the implementation follow the approved plan? Are there deviations? If so, are they justified improvements or oversights?
2. **Bugs**: Are there logic errors, off-by-one errors, null pointer risks, race conditions, or other bugs?
3. **Error Handling**: Are errors handled appropriately? Are there silent failures or swallowed exceptions?
4. **Test Coverage**: Are there tests? Do they cover the main paths and edge cases from the plan?
5. **Security**: Are there injection risks, authentication bypasses, data leaks, or other OWASP Top 10 issues?
6. **Code Quality**: Is the code readable, well-structured, and maintainable? Are there unnecessary complexities?

## Output Format

Write your review as structured markdown. For each finding, use a severity tag:

- `[CRITICAL-N]` — Must be fixed. Bugs, security issues, or major plan deviations.
- `[HIGH-N]` — Should be fixed. Significant quality or correctness concerns.
- `[MEDIUM-N]` — Worth addressing. Improvement opportunity.
- `[LOW-N]` — Minor suggestion. Nice to have.

If there are no findings, explicitly write `No findings.` and still include numeric Summary counts for Critical/High/Medium/Low.
